# 설계 결정 기록

중요한 기술 선택과 사용자 확인 결과를 기록한다. 결정되지 않은 항목을 임의로 확정하지 않는다.

## 2026-07-19 기술 스택 결정

사용자가 앱 제작 시작을 승인함에 따라 직전 제안한 권장 구성을 기준으로 다음 기술 스택을 채택했다.

- 프런트엔드: React 19 + Vite + TypeScript
- 백엔드: NestJS 11 + TypeScript
- WebSocket: Socket.IO
- 데이터베이스와 ORM: PostgreSQL 17 + Prisma
- 패키지 관리: pnpm workspace 모노레포
- 로컬 데이터베이스: Docker Compose

프런트와 서버의 언어를 TypeScript로 통일해 이벤트·데이터 타입을 공유하고, 사용자가 한 언어로 전체 프로그램을 분석할 수 있게 한다.

### 현재 확정된 제품 범위

- 초기 운영 화면은 관리자 페이지와 Agent 페이지로 나눈다.
- 관리자는 Agent, 호텔, 룸을 추가하고 호텔 필터로 룸 테이블을 조회한다.
- 룸별 QR 생성·확인·다운로드·삭제·재생성은 MVP 이후다. (당시 결정이며 2026-07-22 고정 QR 결정으로 대체됨)
- MVP 투숙객 흐름은 테스트용 링크 또는 접근 키를 사용한다.
- 상담은 서버 기준 최대 15분이며 종료 후 메시지를 차단한다.

## 2026-07-19 Phase 1 세션 시간 결정

- MVP에서는 `ChatSession.createdAt`을 상담 제한 시간의 시작으로 보고 생성 시 `expiresAt = 생성 시각 + 15분`을 저장한다.
- Agent가 수락할 때 `startedAt`을 별도로 기록하지만 `expiresAt`을 연장하지 않는다.
- 대기 시간과 실제 상담 시간을 각각 별도로 제공하려면 원본 사양서에 대기 만료 시간을 추가로 정의해야 하므로 현재 범위에는 넣지 않는다.
- 동일 룸에는 `WAITING` 또는 `ACTIVE` 상담을 동시에 하나만 허용한다.
- 투숙객 접근 키 검증 JWT는 상담 생성에만 사용하고, 생성 후에는 DB에 해시로 저장한 별도 불투명 토큰으로 상담을 조회한다.

## 2026-07-19 Phase 2 실시간 통신 결정

- Socket.IO 네임스페이스는 `/chat`으로 고정한다.
- 직원은 `staffToken`, 투숙객은 `guestToken + sessionId`를 handshake auth에 전달한다.
- 비동기 인증 경쟁을 막기 위해 Gateway 연결 이벤트가 아니라 Socket.IO 선행 미들웨어에서 인증한다.
- 상담방 이름은 서버 내부에서 `session:{sessionId}`로 만들며 클라이언트가 임의 방 이름을 지정할 수 없다.
- 메시지는 PostgreSQL 저장 성공 후 승인·상대방 전달 순서로 처리한다.
- 중복 메시지는 `sessionId + clientMessageId` 고유 제약으로 판별하며 기존 메시지 승인만 반환한다.
- Socket.IO 재연결 후 REST 메시지 조회로 연결 단절 구간을 복구한다.

## 2026-07-19 Phase 3 화면 결정

- 실제 QR 도입 전까지 투숙객 링크는 `?accessKey=...`를 사용하고 서버 검증 실패 시 별도 오류 화면을 표시한다.
- 신규 상담 생성 전 언어 선택과 저장 안내 동의를 필수로 받는다.
- 투숙객 세션과 Agent 로그인은 탭 단위 `sessionStorage`에 저장하며 새로고침 때 복구한다. (당시 결정이며 2026-07-22 동일 기기 QR 재접속 결정으로 투숙객 저장 정책만 대체됨)
- Agent 목록은 5초 폴링하고 채팅 메시지와 세션 상태는 Socket.IO로 즉시 갱신한다.
- 종료 작업은 브라우저 기본 확인창 대신 접근 가능한 화면 내 대화상자를 사용한다.
- 한 객실의 열린 상담은 클라이언트 가드만 신뢰하지 않고 PostgreSQL 부분 고유 인덱스로 최종 보장한다.

## 2026-07-21 24시간 인증과 모바일 UX 결정

- Agent·관리자 로그인은 사용자 확인에 따라 `localStorage`나 쿠키가 아닌 탭 단위 `sessionStorage`에 저장한다.
- 콜센터 PC가 24시간 상시 켜지는 운영 조건에 따라 직원 JWT에는 만료 시각을 넣지 않는다.
- 무기한 JWT의 계정 폐기 위험을 줄이기 위해 모든 직원 REST 요청과 WebSocket 연결에서 DB의 계정 존재·활성 상태·역할을 다시 확인한다.
- 투숙객 상담 토큰은 객실·탭 간 격리를 위해 기존 `sessionStorage`와 서버의 15분 상담 만료 정책을 유지한다. (당시 결정이며 2026-07-22 동일 기기 QR 재접속 결정으로 대체됨)
- UI 언어 선택만 비민감 환경설정으로 `localStorage`에 저장한다. 현재 광고·분석·추적 쿠키는 사용하지 않는다.
- 채팅 자동 스크롤은 전체 문서가 아니라 메시지 컨테이너에만 적용하고, 상담 목록 복귀 시 기존 스크롤 위치를 복원한다.
- 모바일 상담 목록과 관리 표는 가로 스크롤 표가 아니라 항목명이 보이는 카드형 목록으로 제공한다.

## 2026-07-22 동일 기기 QR 재접속 결정

- 투숙객 상담의 세션 ID·불투명 토큰·만료 시각은 객실 접근 키별 `localStorage`에 저장한다.
- 같은 기기에서 QR을 다시 읽으면 저장 토큰으로 서버 상담 상세를 재검증하고 `WAITING` 또는 `ACTIVE`일 때만 기존 채팅으로 복귀한다.
- 브라우저 시각상 만료, 서버의 `CLOSED`·`EXPIRED`, 401·403·404 인증/조회 거부 시 해당 객실 복구 정보만 삭제한다.
- 일시적인 네트워크 또는 서버 오류에는 복구 토큰을 삭제하지 않아 장애 복구 후 다시 시도할 수 있게 한다.
- 서버의 15분 만료와 토큰 해시 검증은 그대로 유지하며, 이 기능은 동일 브라우저 기기 복구만 제공하고 다른 기기 복구는 제공하지 않는다.

## 2026-07-19 Phase 3A~5 운영 결정

- 관리자 MVP는 로그인, Agent 추가·목록, 호텔 추가, 룸 추가·필터까지만 포함하며 QR 버튼은 비활성 확장 자리로 둔다. (당시 결정이며 2026-07-22 고정 QR 결정으로 대체됨)
- Agent 생성 비밀번호는 영문과 숫자를 포함한 8자 이상으로 제한하고 bcrypt cost 12로 저장한다. (당시 결정이며 2026-07-22 무료 테스트 비밀번호 결정으로 대체됨)
- 자동 만료는 서버 시작 직후와 5초 주기로 실행해 재시작 중 누락된 상담도 복구한다.
- 단일 서버 MVP의 REST 요청 제한은 메모리에 저장한다. 다중 서버 배포 전에는 Redis 기반 공용 제한기로 교체한다.
- React의 기본 텍스트 이스케이프와 서버의 문자열·길이·형식 검증을 함께 사용하며 HTML 메시지 유형은 제공하지 않는다.
# 2026-07-21 — 전체 Docker 구성과 룸 고객 주소

- PostgreSQL만 실행하던 Compose를 서버, 관리자/Agent 웹, 고객 웹, 마이그레이션까지 포함하는 전체 스택으로 확장한다.
- 룸 생성 시 접근키를 자동 발급하고 관리자 룸 목록에 고객 주소 열기·복사 기능을 제공한다.
- 접근키는 검증용 SHA-256 해시와 복원용 AES-256-GCM 암호문으로 나누어 저장한다. 복호화 비밀값은 `ACCESS_KEY_ENCRYPTION_SECRET`으로 관리한다.
- QR 생성·다운로드·폐기·재발급은 원본 사양대로 MVP 이후 기능으로 유지한다. (당시 결정이며 2026-07-22 고정 QR 결정으로 대체됨)

## 2026-07-22 객실 고정 QR 운영 결정

- 객실에 인쇄하여 장기간 비치할 목적에 맞춰 QR은 룸의 현재 고객 URL을 그대로 담고 정기 만료나 갱신을 두지 않는다.
- 관리자는 룸 목록에서 QR을 미리 보고 약 1024px 고해상도 PNG로 다운로드할 수 있다.
- QR 이미지는 관리자 브라우저에서 필요할 때 생성하며 별도 QR 데이터나 이미지 파일을 서버·DB에 저장하지 않는다.
- 폐기·재발급 UI는 제공하지 않는다. 객실 또는 접근 키를 삭제하면 기존 QR도 사용할 수 없어진다.
- 고객 웹 도메인 변경은 인쇄된 QR을 무효화하므로 상업 인쇄 전에 운영 도메인을 확정하고 실제 휴대전화 스캔 시험을 한다.
- QR 생성에는 MIT 라이선스의 `qrcode` 1.5.4를 사용하고 배포 고지에 저작권·라이선스 정보를 유지한다.

## 2026-07-22 무료 테스트 계정 비밀번호 결정

- 사용자 결정에 따라 무료 테스트 배포의 관리자 계정은 `admin / admin`, Agent 계정은 `agent01 / agent01`로 고정한다.
- 로그인과 관리자 Agent 생성의 비밀번호 길이·문자 조합 제한을 제거한다. 로그인 ID·이름 등 비밀번호가 아닌 필드의 검증은 유지한다.
- 비밀번호가 단순하더라도 DB에는 평문이 아닌 bcrypt cost 12 해시만 저장한다.
- 기존 배포 DB 계정은 일회성 데이터 마이그레이션으로 변경하고, 새 테스트 환경은 명시적인 취약 비밀번호 허용 플래그가 있을 때만 같은 시드 값을 사용한다.
- 당시 무료 테스트에서는 서버 시작 시 두 고정 계정의 bcrypt 해시를 다시 맞추기로 했으나, 아래의 직원 비밀번호 변경 결정에서 변경값 보존을 위해 이 동작을 폐기했다.
- 이 계정은 공개 상업 서비스에서 사용할 수 없으며 상업 전환 전에 강한 개별 계정과 로그인 보호 정책으로 교체한다.

## 2026-07-22 직원 비밀번호 변경 결정

- 관리자와 Agent는 로그인 후 현재 비밀번호를 확인받아 자기 계정의 비밀번호를 변경할 수 있다. 기본 테스트 비밀번호 자체는 사용자가 변경하기 전까지 유지한다.
- 변경 시 계정 `tokenVersion`을 증가시키고 REST·WebSocket 인증에서 JWT 버전과 DB 버전을 비교해 다른 PC를 포함한 기존 무기한 토큰을 모두 폐기한다.
- 위의 서버 시작 강제 비밀번호 재설정 결정은 이 기능이 변경값을 보존할 수 있도록 폐기한다. 시드는 새 계정 생성에만 기본값을 사용하고 기존 계정 비밀번호를 덮어쓰지 않는다.
- 비밀번호 길이·문자 조합 제한을 새로 추가하지 않으며, 브라우저가 비밀번호 변경을 인식할 수 있도록 표준 자동완성 의미를 사용한다.

## 2026-07-22 무료 테스트 계정 복구 결정

- 기존 Render DB에서 과거 내용으로 이미 실행된 마이그레이션은 같은 파일을 수정해도 Prisma가 재실행하지 않으므로 새 일회성 마이그레이션으로 `admin / admin`, `agent01 / agent01`을 복구한다.
- 첫 복구 파일도 Render 적용 이력과 충돌해 실제 DB에 이전 비밀번호가 남은 것을 배포 검증으로 확인했으므로, 더 최신 이름의 최종 일회성 마이그레이션을 추가해 두 계정 해시를 조건 없이 한 번 갱신한다.
- 실제 해시가 다른 경우에만 변경하고 `tokenVersion`을 함께 증가시켜 이전 비밀번호로 발급된 무기한 JWT를 폐기한다.
- 시작할 때마다 비밀번호를 덮어쓰는 동작은 다시 만들지 않으며 이번 복구 뒤에는 본인 비밀번호 변경값을 유지한다.

## 2026-07-22 배포 후 비밀번호 재초기화 방지 결정

- Render 코드 배포 뒤 과거 시드 값으로 되돌아가는 현상이 확인되어 `SEED_RESET_EXISTING_PASSWORDS` 기능 자체를 제거한다.
- 시드는 신규 계정 생성만 담당하고 기존 `admin`, `agent01`을 포함한 모든 직원 계정의 비밀번호를 갱신하지 않는다. 기본 비밀번호 복구가 필요하면 자동 배포가 아니라 본인 비밀번호 변경 또는 명시적인 운영 절차를 사용한다.
- Render 무료 플랜에는 pre-deploy 명령이 없으므로 마이그레이션은 안전을 위해 기존 시작 명령에 유지한다. 로그인 화면 진입 시 헬스 요청 1회로 실제 사용자 요청에 한해 기동을 앞당기며 무료 인스턴스를 상시 깨우는 핑은 사용하지 않는다.

## 2026-07-22 공동 로그 보안·부하 결정

- 공개 상담 직렬화 경계를 REST와 WebSocket 이벤트에 공통 적용해 투숙객 토큰 해시와 직원 비밀번호 해시·토큰 버전을 제거하고 담당 직원은 `id`, `name`만 반환한다.
- Agent와 관리자의 완료 로그 상세는 실시간 방에 접속하지 않는 읽기 전용 모달로 통일한다.
- 5초 폴링은 `OPEN` 상담만 조회하고 30일치 `COMPLETED` 로그는 최초 진입·상태 변경·60초 주기에만 조회한다.
- 직원 목록 GET의 요청 제한 키는 `IP + Bearer 토큰 SHA-256 지문`으로 분리해 한 콜센터 NAT의 여러 로그인 세션이 서로의 분당 한도를 소진하지 않게 하며 원본 JWT는 메모리에 보관하지 않는다.

## 2026-07-22 상담 기록 30일 보존 결정

- `CLOSED`, `EXPIRED`, `CANCELLED`, `BLOCKED` 상담은 `closedAt`부터 30일이 지나면 세션과 모든 메시지를 영구 삭제한다.
- `WAITING`, `ACTIVE` 상담과 종료 시각이 없는 데이터는 잘못 삭제하지 않는다.
- 무료 Vercel·Render 플랜을 고려해 요청마다 정리하지 않고 서버 시작 후 1회와 24시간마다 1회만 인덱스 기반 삭제를 실행한다.
- 이 삭제는 복구할 수 없으므로 개인정보 처리방침에 30일 보존 기간을 고지하고 운영 백업도 같은 보존 정책과 충돌하지 않게 구성한다.

## 2026-07-22 Agent 수신 팝업 알림 결정

- 새 WAITING 상담은 단순 개수 증가가 아니라 상담 ID 집합 차이로 감지해 대기 수가 0에서 1로 바뀌는 첫 상담도 놓치지 않는다.
- 새 상담과 해당 Agent에게 배정된 모든 진행 상담의 고객 메시지는 화면 우측 상단 팝업으로 제공하며 팝업은 8초 후 자동으로 닫힌다. 알림음은 기본 꺼짐이고 Agent가 직접 켜거나 끌 수 있으며 같은 브라우저에 선택을 저장한다.
- 새 상담 팝업의 확인 동작은 즉시 수락하지 않고 대기 목록으로 이동시켜 여러 Agent의 동시 수락 충돌을 피한다.
- 브라우저 시스템 알림은 사용자가 버튼으로 권한을 허용한 뒤 탭이 백그라운드일 때만 추가로 표시한다. 권한이 없어도 앱 내부 팝업은 동작한다.
- 새 상담·고객 메시지가 비활성 탭에 도착하면 탭 제목을 깜빡이며, 사용자가 다시 Agent 창을 보면 읽음 동작으로 간주해 원래 제목으로 복구한다.
- 진행 상담 목록은 세션 생성순이 아니라 마지막 메시지 시각 기준으로 정렬해 새 메시지가 온 상담을 맨 위로 이동한다.
- Vercel·Render 무료 플랜에서는 목록 폴링마다 메시지를 집계하지 않는다. 메시지 저장 때 `ChatSession.lastActivityAt`만 함께 갱신하고 인덱스로 정렬해 DB 부하를 제한한다.
