# 상업 출시 전 점검 체크리스트

마지막 점검: 2026-07-21

이 문서는 코드와 현재 테스트 배포 구성을 기준으로 한 실무 점검표이며 법률 자문이 아니다. 실제 서비스 국가, 계약 주체, 호텔과의 데이터 처리 관계가 확정되면 일본 개인정보보호법(APPI) 등 적용 법률에 관해 전문가 검토를 받아야 한다.

## 1. 현재 라이선스 판단

- [x] 프로젝트 루트에 MIT `LICENSE`가 있으며 상업적 사용·수정·배포를 허용한다.
- [x] 현재 production 의존성 잠금 파일에서 GPL·AGPL·LGPL·SSPL 계열은 발견되지 않았다.
- [x] 직접 의존성은 MIT, Apache-2.0, BSD 계열 중심으로 상업 서비스 사용이 가능하다.
- [x] `THIRD_PARTY_NOTICES.md`에 주요 패키지와 CC-BY-4.0 데이터 출처를 기록했다.
- [ ] 상업 출시 주체가 현재 `LICENSE`의 저작권자 `Sangoon`과 일치하는지 확인하고, 공동 개발자의 코드·디자인·브랜드 사용 권한을 서면으로 정리한다.
- [ ] 출시 직전 `pnpm licenses:list`를 다시 실행하고 새 의존성, 각 패키지의 라이선스 전문과 Apache `NOTICE` 요구사항을 배포 산출물에 반영한다.
- [ ] `REMOTE+` 명칭·로고·호텔 이미지·글꼴 등 코드 외 자산의 상표권과 상업 이용 권한을 별도로 확인한다.

MIT는 상업 이용을 허용하지만 저작권 표시와 허가 문구를 복제물 또는 중요한 부분에 유지하도록 요구한다. Apache-2.0 구성요소를 재배포할 때는 라이선스 사본과 원본에 `NOTICE`가 있다면 해당 고지를 유지해야 한다. CC-BY-4.0 자료는 적절한 출처와 라이선스 링크가 필요하다.

## 2. 개인정보와 브라우저 저장소

현재 처리 가능 정보에는 상담 메시지, 호텔·객실 식별자, 직원 계정, 담당 상담 기록, 접속 IP가 포함된다. 고객이 메시지에 이름·전화번호·건강정보 등을 입력할 가능성도 있으므로 단순 익명 채팅으로 가정하면 안 된다.

- [x] 직원·관리자·고객 인증은 탭 단위 `sessionStorage`를 사용한다.
- [x] UI 언어만 `localStorage`에 저장한다.
- [x] 현재 광고·분석·추적 쿠키와 제3자 분석 SDK는 사용하지 않는다.
- [ ] 고객 화면에 사업자명, 이용 목적, 저장 항목, 보존 기간, 처리 위탁사, 문의·삭제 요청 방법을 포함한 개인정보 처리방침 링크를 제공한다.
- [ ] 호텔, 중앙 콜센터 운영사, RemotePlus 중 누가 개인정보 처리 책임자이고 누가 수탁자인지 계약서에서 확정한다.
- [ ] 상담 메시지와 로그의 보존 기간 및 자동 삭제 작업을 구현한다. 현재 자동 보존 만료가 없어 상업 출시 차단 항목이다.
- [ ] Render Singapore DB를 계속 사용한다면 국외 처리·위탁 여부와 제공자 계약을 검토한다. Sakura VPS 일본 리전으로 옮겨도 백업·로그·지원 업체의 실제 처리 위치를 함께 확인한다.
- [ ] 개인정보 열람·정정·삭제·이용정지 요청 처리 절차와 유출 사고 보고 절차를 마련한다.
- [ ] 향후 분석·광고·외부 채팅 SDK를 추가하면 쿠키 동의뿐 아니라 일본 전기통신사업법의 외부송신 규율 적용 여부도 다시 검토한다.

일본 개인정보보호위원회는 이용 목적의 통지·공표, 안전관리조치, 국외 제3자 제공 시 정보 제공과 동의 등에 관한 지침을 제공한다.

- [개인정보보호법 통칙 가이드라인](https://www.ppc.go.jp/personalinfo/legal/guidelines_tsusoku/)
- [외국에 있는 제3자 제공 가이드라인](https://www.ppc.go.jp/personalinfo/legal/guidelines_offshore/)

## 3. 인증과 운영 보안

- [x] 직원 JWT는 24시간 PC 운영을 위해 만료 시각이 없지만 모든 REST·WebSocket 인증에서 계정 활성 상태와 역할을 DB로 재확인한다.
- [x] 브라우저 탭 종료와 명시적 로그아웃 시 직원 인증이 제거된다.
- [ ] 무기한 토큰을 강제 폐기할 수 있는 관리자용 전체 로그아웃 또는 계정별 토큰 버전 기능을 구현한다. JWT가 유출되면 계정 삭제 외에는 즉시 회수 수단이 없다.
- [ ] 여러 사람이 하나의 직원 ID를 공유하면 실제 처리자를 감사 로그에서 구분할 수 없다. 상업 운영 전 개인별 계정 또는 교대 근무자 식별 절차를 결정한다.
- [ ] 운영 비밀번호 정책, MFA, 로그인 실패 잠금, 비밀값 교체 절차를 마련한다.
- [ ] PostgreSQL 자동 백업, 복구 시험, 장애 알림, 중앙 로그와 모니터링을 구성한다.
- [ ] 무료 Render DB 만료 전에 유료 DB 또는 Sakura VPS 운영 DB로 이전한다.
- [ ] 취약점·의존성 업데이트 정책과 정기 보안 점검 일정을 만든다.

## 4. 서비스 계약과 고객 고지

- [ ] 이용약관에 서비스 범위, 금지행위, 책임 제한, 장애·점검, 준거법, 해지 정책을 명시한다.
- [ ] 호텔별 서비스 수준, 장애 연락망, 데이터 처리 위탁, 보안 책임과 사고 통지 기한을 계약한다.
- [ ] 고객이 비밀번호·결제정보·민감정보를 입력하지 않도록 현재 안내를 유지하고, 실제 개인정보 처리방침 링크를 추가한다.
- [ ] 결제 기능을 추가할 경우 카드 정보를 직접 저장하지 않고 적격 결제대행사를 사용한다.

## 현재 판단

현재 테스트 배포를 계속 검증하는 데 직접적인 오픈소스 라이선스 차단 요소는 발견되지 않았다. 그러나 상업 출시 전에는 개인정보 처리방침·보존 기간 자동 삭제·운영 백업·개별 작업자 감사 추적·무기한 토큰 회수 수단을 필수 보완 항목으로 본다.
